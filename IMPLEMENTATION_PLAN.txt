SECTION 1: THE THREE-LAYER ARCHITECTURE
================================================================================

This application is built using a strict three-layer model. Each language has
a clearly defined responsibility and they never cross lanes.

LAYER 1 - C Backend (c_programs/backend.c)
-------------------------------------------
  ROLE: The "Brain" - All core business logic and data management lives here.

  WHY C?
  C gives direct control over RAM. All data (vegetables, orders) is stored
  in C data structures (arrays, linked lists) during runtime instead of
  hitting disk on every operation. This is dramatically faster than reading
  and writing files constantly.

  WHAT IT DOES:
  - Loads inventory and orders from .txt files into memory on startup.
  - Stores vegetables in a fixed global array:
      Item g_inventory[50];   // Each slot = name, price, stock, unit
  - Stores customer orders in a modern FIFO Queue (First-In, First-Out):
      Order *g_orders_head;   // Points to the oldest order (front of line)
      Order *g_orders_tail;   // Points to the newest order (end of line)
  - Uses optimized helper functions (process_items, split, find_order) to 
    handle complex logic like inventory restoration and item parsing.
  - Processes COMMANDS sent by Python over stdin/stdout (see Section 2).
  - On graceful shutdown (Ctrl+C), saves all in-memory data back to .txt files.

  KEY FUNCTIONS:
  - load_inventory()     : Parses data/inventory.txt into g_inventory[]
  - load_orders()        : Parses data/orders.txt into the g_orders linked list
  - save_data()          : Writes both back to disk (only called at shutdown)
  - cmd_place_order()    : Creates a new OrderNode, deducts stock from g_inventory
  - cmd_update_status()  : Updates Order.status; if "Cancelled", refunds stock
  - cmd_pay_order()      : Marks an order as Paid
  - cmd_list_inventory() : Dumps the g_inventory array back to Python
  - handle_sigint()      : Signal handler - intercepts Ctrl+C, calls save_data()

  DATA STRUCTURES IN DETAIL:
  
    typedef struct {
        char name[50];    // "Tomato"
        int  price;       // 40
        double stock;     // 38.5
        char unit[20];    // "kg"
    } Item;

    typedef struct OrderNode {
        char id[30];                  // "#OD-00001"
        char name[100];               // "John Doe"
        char phone[20];               // "9876543210"
        char email[100];              // "john@example.com"
        char address[200];            // "Flat 10, XYZ Street..."
        char payment[20];             // "upi" | "card" | "cod"
        char items_str[2048];         // "Tomato:2.5:kg:40;Onion:1:kg:29;"
        double subtotal;              // 130.00
        double total;                 // 145.00
        char status[30];              // "Pending" | "Paid" | "Packed"...
        char date[30];                // "18-04-2026"
        struct OrderNode *next;       // Pointer to the next order in the list
    } Order;

LAYER 2 - Python Flask (app.py)
---------------------------------
  ROLE: The "Connector" - Routes HTTP requests, talks to C, sends emails.

  WHY PYTHON?
  Flask is ideal for quickly building web routes and handling HTTP. Python
  also has excellent libraries for sending emails (smtplib) which would be
  complex to implement in C.

  WHAT IT DOES:
  - Starts the compiled C backend (backend.exe) as a background subprocess
    using Python's subprocess module with stdin/stdout pipes.
  - Every time a user clicks something on the website, Flask receives the
    HTTP request, assembles a text COMMAND string, and sends it to the
    C backend through the subprocess pipe.
  - It reads the C backend's text RESPONSE back through stdout and parses
    key-value pairs (e.g. "NAME:John\nPHONE:999\nEND").
  - It passes parsed data to Jinja2 templates to render the final HTML.
  - It sends emails directly using Gmail SMTP (smtplib) when payment is
    confirmed or when an admin updates an order status.

  KEY ROUTES:
  - GET  /              : Home page (fetches live inventory from C)
  - GET  /order         : Order form (fetches inventory)
  - POST /order         : Submits order (sends PLACE_ORDER cmd to C)
  - GET  /bill          : Bill page (fetches specific order from C)
  - POST /payment       : Payment (sends PAY_ORDER cmd to C, triggers email)
  - POST /cancel_order  : Cancel (sends UPDATE_STATUS "Cancelled" to C)
  - GET  /admin         : Admin dashboard (fetches all orders from C)
  - POST /admin/update_status  : Updates order status (triggers status email)
  - POST /admin/update_inventory : Updates vegetable stock in C memory
  - POST /admin/add_vegetable   : Adds a new vegetable to C memory

LAYER 3 - HTML/CSS/JavaScript (templates/ + static/)
-------------------------------------------------------
  ROLE: The "Face" - Everything the user sees and interacts with.

  WHAT IT DOES:
  - Uses Flask's Jinja2 template engine to receive data from Python
    and render it into HTML dynamically (e.g. {{ veg.name }}, {{ data.TOTAL }}).
  - Contains vanilla JavaScript for client-side UI logic:
      * Auto-formats card number as xxxx-xxxx-xxxx-xxxx (with hyphens)
      * Auto-formats expiry date as MM/YY (with smart month padding)
      * Validates CVV length and card number length in real-time
      * Forces cardholder name to uppercase
      * Calculates live order subtotal as user selects quantities
      * Validates minimum Rs. 100 order value before form submission
  - Inherits a shared layout from templates/base.html via Jinja2 extends.
  - Static assets (styles.css, upi_qr.png) sit in static/ and are served
    directly by Flask.

SECTION 2: HOW DATA IS STORED IN C MEMORY (RAM)
================================================================================

In our stateful C backend architecture, we completely avoid reading and writing
to text files every time a user clicks a button. Instead, when the program
boots up, it reads the files ONCE and loads all the information tightly into
the CPU's RAM using standard C data structures.

Here is exactly how the three core pieces of data are stored in C memory:

  2a. INVENTORY DATABASE - g_inventory[] Array
  ---------------------------------------------
  All vegetables are loaded into a fixed-size global Array of Structs.
  A custom Item struct holds all the specs for a single vegetable:

      typedef struct {
          char   name[50];    // e.g. "Spinach"
          int    price;       // e.g. 30
          double stock;       // e.g. 15.00
          char   unit[20];    // e.g. "bunches"
      } Item;

      // Physically allocates space in RAM for up to 50 vegetables
      static Item g_inventory[MAX_VEGS];

  Because it is an array, when the front page or checkout screen loads, the
  C code can loop through all 50 slots in under a microsecond and shoot the
  live stock data over to the Python Flask server.

  2b. LIVE ORDERS DATABASE - g_orders FIFO Queue
  -------------------------------------------------
  To ensure orders are processed in the order they were received, the system
  implements a First-In-First-Out (FIFO) Queue using two anchor pointers:
  
      typedef struct OrderNode {
          char id[30];                  // "#OD-00001"
          char name[100];               // "John Doe"
          char phone[20];               // "9876543210"
          char email[100];              // "john@example.com"
          char address[200];            // "Flat 10, XYZ Street..."
          char payment[20];             // "cod", "upi", "card"
          char items_str[MAX_ITEMS_STR];// "Tomato:5.0:kg:50;Onion:1:kg:30;"
          double subtotal;              // 130.00
          double total;                 // 145.00
          char status[30];              // "Pending", "Cancelled", "Packed"
          char date[30];                // "18-04-2026"
          struct OrderNode *next;       // Pointer to the next Order in RAM
      } Order;

      static Order *g_orders_head = NULL; // Points to the oldest order
      static Order *g_orders_tail = NULL; // Points to the newest order

  THE QUEUE ADVANTAGE:
  - ENQUEUE: When a new order is placed, we use the g_orders_tail pointer to
    instantly attach it to the end of the line (O(1) complexity).
  - PERSISTENCE: When saving to disk, we start at the head and walk to the 
    tail, ensuring the oldest orders are always at the top of the .txt file.
  - ACCURACY: There is zero risk of "reversing" the order history during 
    saving/loading, which was a bug in the old stack-based list.

  3. CODE OPTIMIZATION HELPER FUNCTIONS
  --------------------------------------
  The current version of backend.c utilizes high-efficiency helpers to 
  keep the code clean and explainable:

  - split(): Replaces complex pointer loops with a single function that 
    breaks lines by ':' or '|' into easy-to-read arrays.
  - find_order(): A single reusable search engine for the linked list.
  - process_items(): A "Swiss-army knife" function. It parses the order 
    string and can either PRINT the items (for bill receipts) or RESTORE 
    stock to inventory (for cancellations).

  2c. ADMIN CREDENTIALS - g_admin_user / g_admin_pass
  -----------------------------------------------------
  Since there is only one set of master credentials for the admin dashboard,
  no arrays or linked lists are needed. On boot, the values are read from
  admins.txt and cached into two simple C character arrays:

      static char g_admin_user[50];
      static char g_admin_pass[50];

  THE CORE RUNTIME LOOP:
  As the web server runs, modifying stock or cancelling orders simply tweaks
  fields inside these exact structs, living purely in volatile RAM. When you
  eventually press Ctrl+C to shut down, the C backend safely traverses the
  Array and the Linked List, converts them back into formatted strings, and
  rewrites inventory.txt and orders.txt for the next session.

SECTION 3: HOW THE THREE LANGUAGES COMMUNICATE
================================================================================

The core communication mechanism is a PERSISTENT SUBPROCESS PIPE.

Here is the exact step-by-step flow for a customer placing an order:

STEP 1 - SERVER BOOT
  Python (app.py) runs: subprocess.Popen(["bin/backend.exe"], 
                                          stdin=PIPE, stdout=PIPE)
  The C backend wakes up, loads inventory.txt and orders.txt into RAM,
  and then enters an infinite read loop: while(fgets(line, stdin)) { ... }
  It is now waiting for commands from Python.

STEP 2 - USER FILLS ORDER FORM
  The Browser sends an HTTP POST request to Python Flask at /order.
  Flask reads the form data: name, phone, address, vegetable quantities.

STEP 3 - PYTHON ASSEMBLES A COMMAND
  Flask builds a text command string:
    cmd = 'PLACE_ORDER "John" "9876543210" "john@email.com" "Flat 10" 
           "cod" "Tomato:2.5:40;Onion:1:29;"'

STEP 4 - PYTHON WRITES TO C'S STDIN
  Flask writes this string through the subprocess pipe:
    process.stdin.write(cmd.encode() + b'\n')
    process.stdin.flush()

STEP 5 - C BACKEND READS AND PROCESSES
  The C backend's read loop picks up the command line.
  It tokenizes it and dispatches to cmd_place_order():
  - Allocates a new OrderNode on the heap using malloc()
  - Fills in all the order fields
  - Deducts quantities from g_inventory[]
  - Generates a new Order ID (#OD-XXXXX)
  - Appends the node to the g_orders linked list

STEP 6 - C SENDS RESPONSE BACK
  C backend writes a structured text response to stdout:
    printf("SUCCESS\nORDER_ID:#OD-00001\nNAME:John\nTOTAL:145.00\nEND\n");

STEP 7 - PYTHON READS AND PARSES
  Flask reads lines from the subprocess stdout until it sees "END".
  It parses each "KEY:VALUE" line into a Python dictionary.

STEP 8 - FLASK RENDERS HTML
  Flask passes the parsed dictionary to Jinja2:
    return render_template("order_success.html", data=parsed_dict)
  Jinja2 replaces {{ data.ORDER_ID }} with "#OD-00001" etc.
  The final HTML page is sent back to the customer's browser.

THE RESULT: The browser displays a clean, formatted order success screen
            with the exact data that was computed and stored by C in RAM.

SECTION 3: THE DATA PERSISTENCE LIFECYCLE
================================================================================

  [SERVER START]
       |
       v
  C reads inventory.txt + orders.txt into RAM structs
       |
       v
  [RUNTIME: All operations happen in RAM only]
  - Placing orders modifies g_inventory[] and g_orders linked list
  - Updating status modifies Order.status field
  - Cancelling restock modifies g_inventory[].stock
  - NOTHING is written to disk during this phase
       |
       v
  [SHUTDOWN: User presses Ctrl+C]
       |
       v
  Python's atexit handler sends "SHUTDOWN" command to C
       |
       v
  C intercepts SIGINT signal via handle_sigint()
  C calls save_data() which:
  - Loops through g_inventory[] and writes each Item to inventory.txt
  - Traverses the g_orders linked list and writes each Order to orders.txt
       |
       v
  C frees all malloc'd memory, calls exit(0)
  Python process also exits cleanly.

SECTION 4: THE EMAIL NOTIFICATION SYSTEM
================================================================================

  WHO SENDS EMAILS? Python (app.py) exclusively. C has NO email logic.

  HOW:
  Python uses the smtplib library to connect to Gmail's SMTP server.
  A Gmail App Password (not the regular password) is used for authentication.

  WHEN ARE EMAILS SENT?
  1. PAYMENT CONFIRMED: When a customer completes payment via /payment
     Python calls email_payment_confirmed(name, email, order_id, method, total)

  2. ORDER STATUS UPDATE: When an admin changes order status in the dashboard
     Python calls the appropriate email function based on the new status:
     - "Packed"     -> email_order_packed()
     - "Dispatched" -> email_order_dispatched()
     - "Delivered"  -> email_order_delivered()
     - "Cancelled"  -> email_order_cancelled()

  NOTE: No email is sent when an order is first placed or when a customer
        cancels before payment. Only paid orders trigger status emails.

================================================================================
SECTION 5: ADMIN AUTHENTICATION
================================================================================

  The admin login uses a simple session-based system managed by Flask.

  - Admin credentials (username, password) are stored in data/admins.txt
  - On boot, C reads these into g_admin_user[] and g_admin_pass[] char arrays
  - When admin submits login form, Flask sends LOGIN command to C
  - C compares the submitted credentials against its in-memory values
  - If matched, C responds SUCCESS; Flask sets session["admin"] = True
  - All admin routes check: if not session.get("admin"): redirect(login)
  - Admin can: view all orders, update order status, modify inventory, add veg

================================================================================
SECTION 6: STOCK MANAGEMENT & CANCEL LOGIC
================================================================================

  WHEN AN ORDER IS PLACED:
  - cmd_place_order() loops through the submitted item quantities
  - For each item, it finds the matching entry in g_inventory[]
  - It SUBTRACTS the ordered quantity from g_inventory[i].stock
  - If required quantity > available stock, the order is rejected

  WHEN AN ORDER IS CANCELLED:
  - cmd_update_status() receives ("Cancelled") as the new status
  - It checks: is current status already "Cancelled"? If not, proceed.
  - It parses the order's items_str field: "Tomato:2.5:kg:40;Onion:1:kg:29;"
  - For each item, it ADDS the quantity back to g_inventory[i].stock
  - Then it updates Order.status to "Cancelled"
  - This means stock is automatically refunded on cancellation from BOTH
    the bill page (customer) and the admin dashboard.

SECTION 7: COMPLETE PROJECT DIRECTORY GUIDE
================================================================================

This project is organized into specific folders, each with a critical role:

📁 rewrite/ (Root Directory)
-----------------------------
- app.py         : The CORE of the web server. It handles URL routing, talks
                   to the C backend, and manages email notifications.
- compile.bat    : A helper script for Windows. Double-click to instantly 
                   recompile backend.c into bin/backend.exe.
- requirements.txt : List of libraries (only Flask is needed for this app).

📁 c_programs/
---------------
- backend.c      : The SOURCE CODE for the entire logic. This is where structs,
                   linked lists, file I/O, and command processing live.

📁 bin/ (Binary)
---------------
- backend.exe    : The actual application. This is what Python "runs" in the 
                   background. It is regenerated every time you compile.

📁 data/ (Persistence)
-----------------------
- inventory.txt  : Stores all vegetables. Format: [Name]:[Price]:[Stock]:[Unit]
- orders.txt     : Stores all order history. Uses '|' as a separator.
- payments.txt   : Stores payment confirmation logs.
- admins.txt     : Stores the master username/password.

📁 static/ (Front-end Assets)
------------------------------
- styles.css     : The "Design System". Contains all colors, fonts, and layouts.
- upi_qr.png     : The graphical QR code displayed on the checkout page.

📁 templates/ (Dynamic User Interface)
---------------------------------------
- base.html      : The master template. Controls the navigation bar and footer.
- index.html     : The home page (Shopping catalog).
- order.html     : The "Checkout" screen where users enter quantities.
- bill.html      : The "Payment" portal where users choose UPI/Card/COD.
- order_success.html : The confirmation receipt shown after clicking "Proceed".
- admin.html     : The secret management dashboard with dual-table layout.
- admin_login.html : The secure gateway to the dashboard.
- payment_success.html : The very final screen after a payment is verified.
